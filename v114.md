# Codex v0.114 Hook Research Notes

This note captures the current understanding of Codex `0.114.x` native hook support based on:

- official Codex documentation
- the official `openai/codex` GitHub repository
- the current `0.114.0` release context

The goal is to establish a reliable baseline before comparing OMX `0.9.x` to Chedex and deciding what should be reintroduced, rewritten, or intentionally left out.

## Scope

This is not yet a final design decision document.

It is a grounding memo for:

1. what Codex officially documents today
2. what Codex appears to implement in the official repo
3. what that likely means for a Chedex-native hook strategy
4. which points remain ambiguous and need explicit validation later

## High-Level Takeaway

Codex `0.114.0` still publicly documents `notify` as the main external hook-like integration surface.

However, the official `openai/codex` repository now contains a richer native hooks implementation with typed lifecycle events, notably:

- `SessionStart`
- `Stop`

This suggests the implementation has moved beyond the older `notify`-only model, even if the public docs have not fully caught up yet.

That matters because the main failure mode we care about in Chedex is ambiguous termination. The new `Stop` hook path appears relevant to exactly that problem.

## Officially Documented Surface

### `notify`

The public Codex configuration documentation still describes `notify` as a configured external command in `~/.codex/config.toml`.

Documented behavior:

- Codex invokes the configured command
- Codex sends JSON payload information
- the documented event for external notify is `agent-turn-complete`

The docs also mention TUI notification filtering around:

- `agent-turn-complete`
- `approval-requested`

But the public docs still read as if `notify` is primarily a notification path, not a general lifecycle hook engine.

## Official Repo Implementation Observed

### Dedicated hooks subsystem

The official repository now contains a dedicated hooks crate:

- `codex-rs/hooks`

That alone is a meaningful shift from a simple legacy `notify` model.

### Hook registry/config shape

The repo contains a `HooksConfig`/registry path that includes values such as:

- `legacy_notify_argv`
- `feature_enabled`
- `config_layer_stack`
- `shell_program`
- `shell_args`

This strongly suggests Codex currently supports:

- legacy notification compatibility
- a newer feature-gated hook engine
- shell-based execution for hook commands

Validated locally against the installed `codex-cli 0.114.0` binary:

- the feature flag is `codex_hooks`
- `codex features list` shows `codex_hooks` as an available feature
- hook discovery loads `hooks.json` from config layer folders
- the local binary accepts a Claude-style hook config shape under `hooks.json`

### Typed hook payloads

The repo contains typed payload structures for hook events, including:

- `AfterAgent`
- `AfterToolUse`

The `AfterToolUse` structure appears to include fields such as:

- tool kind
- tool input
- whether the tool executed
- success
- mutating
- sandbox
- sandbox policy
- output preview

This is materially richer than a minimal notification event stream.

### Native lifecycle hook events

The repo hook schema defines event names including:

- `SessionStart`
- `Stop`

This is the most important observation for Chedex.

Codex now provides native lifecycle intervention points for:

- session boot/resume
- end-of-turn / termination control

## `SessionStart` Behavior

From the official source, `SessionStart` receives structured context including:

- session id
- transcript path
- cwd
- model
- permission mode
- source such as `startup` or `resume`

Observed behavior from source:

- plain stdout text can be used as additional model context
- structured hook JSON can add context and affect processing
- malformed JSON-like output is treated as a failure rather than silently accepted

Design implication:

`SessionStart` looks suitable for a recovery / re-grounding harness.

Possible uses in Chedex:

- restore workflow slug
- inject active artifact paths
- inject current phase
- inject `next_step`
- inject unresolved blockers or risks

## `Stop` Behavior

From the official source, `Stop` receives structured information including:

- session id
- turn id
- transcript path
- cwd
- model
- permission mode
- whether a stop hook is already active
- last assistant message

Observed output semantics from source:

- hook output appears to support universal fields such as:
  - `continue`
  - `stopReason`
  - `suppressOutput`
  - `systemMessage`
- stop output appears able to return a blocking decision with a reason
- exit code `2` with stderr text is also treated specially as a stop-block / continue signal

Design implication:

`Stop` looks suitable for a termination gate that prevents ambiguous endings.

Possible uses in Chedex:

- block completion if no terminal status exists
- block completion if verification evidence is missing
- block completion if `progress.json` lacks `next_step` on non-complete states
- force explicit terminal outcomes:
  - `completed`
  - `blocked`
  - `paused`

## Why This Matters For Chedex

Our main concern is not “more automation.”

Our main concern is the ambiguous ending of the main harness:

- work stops without a clean terminal state
- completion is claimed without enough proof
- the session ends without a reliable resume point

The observed Codex native lifecycle hooks appear well aligned to that exact problem.

This means we may be able to solve the real failure mode using Codex-native primitives instead of rebuilding OMX-style external orchestration.

## Likely Best Native Strategy

If the repo implementation is truly available in Codex `0.114.x` user-facing configuration, the best first native strategy would likely be:

### 1. `SessionStart` recovery hook

Purpose:

- recover context after restart, interruption, resume, or compaction

Minimum behavior:

- detect active workflow under `~/.codex/workflows/`
- load compact workflow state
- add minimal continuation context to the session

Targeted first use:

- `autopilot`
- `ralph`

### 2. `Stop` completion gate

Purpose:

- prevent ambiguous or under-evidenced endings

Minimum behavior:

- if the active workflow has no valid terminal state, block stop
- require explicit verification evidence for `completed`
- require an explicit `next_step` or blocker for non-complete states

Targeted first use:

- `ralph`
- `autopilot`
- light variant for `ultrawork`

### 3. keep `notify` as secondary only

Purpose:

- attention routing
- user notification
- optional side-channel integrations

Not recommended as the main lifecycle control surface.

## Minimal Chedex Mapping

If we use native hooks, the clean ownership model would likely be:

- `autopilot`
  - lifecycle owner
  - `SessionStart` rehydrates phase/spec/plan/proof context
  - `Stop` enforces phase-complete terminal state

- `ralph`
  - persistence + execution owner
  - `SessionStart` restores active execution state
  - `Stop` enforces verification and explicit completion / pause / blocker state

- `ultrawork`
  - parallel lane owner
  - little or no heavy recovery logic
  - only a lightweight stop gate to ensure lanes report outcomes cleanly

## Important Ambiguities

These are the parts we should treat as unresolved until we validate them directly:

### 1. Public docs lag vs shipped behavior

The official docs still emphasize `notify`.
The official repo clearly contains a richer hooks subsystem.

This ambiguity is narrower now.

Validated locally:

- the shipped `0.114.0` binary exposes `codex_hooks`
- the shipped binary parses `hooks.json`

Remaining uncertainty:

- how stable OpenAI considers this surface relative to the still-older public docs

### 2. User-facing config syntax

This ambiguity is also narrower now.

Validated locally:

- enable in `config.toml` with `[features] codex_hooks = true`
- provide lifecycle handlers in `hooks.json`
- matcher format is regex-based and accepted by the binary

Still worth treating carefully:

- exact future schema stability
- exact handler ordering guarantees across config layers

### 3. Event coverage

We observed:

- `SessionStart`
- `Stop`
- typed post-agent/post-tool payload structures

Ambiguity:

- which of these are fully wired as public hook events
- whether they are feature-gated
- which are intended for external configuration vs internal use

### 4. Stability contract

The repo implementation is highly relevant, but repo presence alone is not the same as a stable public contract.

Ambiguity:

- what OpenAI considers officially stable vs under-development
- how aggressively Chedex should rely on this feature without broader public documentation

## Working Recommendation For Now

Current updated stance:

- `notify` remains the public-doc baseline
- native `SessionStart` and `Stop` are implemented enough in `0.114.0` to build against
- `hooks.json` plus `codex_hooks = true` is a usable Chedex integration path
- Chedex should still keep the hook layer small and purpose-built around lifecycle governance rather than broad orchestration

## Implemented Direction

Validated and adopted for Chedex:

1. enable `codex_hooks`
2. install a managed `hooks.json`
3. use `SessionStart` for governed workflow rehydration
4. use `Stop` for governed workflow closeout gating
5. keep the external layer limited to workflow state, proof, and recovery
